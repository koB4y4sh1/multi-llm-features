# コーディング規約

このドキュメントは、リポジトリ全体のうち特に `sample/`、`demo/`、`bench/` でコードを追加・変更するときの方針をまとめたもの。

## 基本方針

- 各フォルダの役割を混ぜない
- 実行方法はフォルダの目的に合わせて選ぶ
- import は「書きやすさ」より「実行時に迷いにくいこと」を優先する
- 共通化は最小限に留め、読む人が追いやすい構成を優先する
- 学習用コード、デモコード、計測コードでは最適な構成が違うことを前提にする

## `sample/` の規約

### 役割

- 機能単位の小さなサンプルを置く
- 1 ファイルずつ試せる見本を置く
- 学習用、検証用、API の挙動確認用のコードを置く

### 実装方針

- 各ファイルは、できるだけ単体で読んで理解できる形にする
- 実行の主役になる処理は各ファイルに書く
- プロンプト、モデル呼び出し、出力処理はサンプルごとに明示する
- 共通化するのは定数や軽い補助関数までに留める

### 実行方針

- VS Code の Current File 実行や、ファイル単位の実行と相性がよい構成を優先する
- `launch.json` などで実行環境を補助してよい
- `python -m ...` 前提にはしない

### import 方針

- 絶対 import を基本とする
- `from . import ...` のような相対 import は原則避ける

理由:

- `sample/` は 1 ファイルずつ直接実行することが多い
- 相対 import は module 実行前提になりやすく、初心者がハマりやすい
- 絶対 import の方が import 元を追いやすい

### 具体例

`sample/google/vertext_ai` では、次の分担を採用する。

- `client = genai.Client()` は各サンプルファイルで生成する
- `TEXT_MODEL`、`EMBEDDING_MODEL`、`DATA_DIR`、`read_bytes()` は `sample/google/vertext_ai/__init__.py` に置く
- サンプル側は `from sample.google.vertext_ai import ...` の形で参照する

## `demo/` の規約

### 役割

- ユースケース単位のまとまったデモを置く
- 複数モデル比較や、複数ファイルで構成される実践寄りのコードを置く
- HTML レポートや成果物を出す実行物を置く

### 実装方針

- 単発スクリプトではなく、小さなアプリケーションとして構成する
- 役割ごとにファイルやモジュールを分けてよい
- データセット、出力先、CLI 引数などを明示する
- README に書いた実行コマンドと一致する構成にする

### 実行方針

- `python -m demo....` のような module 実行を基本とする
- 直接ファイル実行よりも、エントリポイントを通した実行を優先する
- README に実行コマンドを明記する

### import 方針

- 絶対 import を基本とする
- package として整理された前提で構わないが、import 規約は統一する
- 相対 import を使う場合は、同一サブパッケージ内で閉じるケースに限定し、混在を増やさない

理由:

- `demo/` は package として実行する方が自然
- 実行方法を `-m` に寄せることで、構成が増えても破綻しにくい
- 絶対 import の方がフォルダ横断の依存関係を追いやすい

## `bench/` の規約

### 役割

- ベンチマーク、比較、評価、計測用のコードを置く
- 再現可能な手順でモデルや設定の差分を見たいときのコードを置く
- 実験結果の確認や、簡易レポート生成の入口となるコードを置く

### 実装方針

- 入力条件、モデル名、評価指標、出力形式をコード上で明示する
- `argparse` などで再実行しやすい CLI を優先する
- 計測ロジックと表示ロジックを可能な範囲で分ける
- 一時的な思いつきコードではなく、あとで比較し直せる形で残す

### 実行方針

- CLI から再現実行できることを優先する
- ファイル直接実行でも `python -m bench....` でもよいが、README やコメントで迷わない形にする
- 出力は比較しやすいテキスト、JSON、HTML などの形を選ぶ

### import 方針

- 絶対 import を基本とする
- 評価ユーティリティや共通レポート処理は既存実装を優先して再利用する
- 計測コードの中にローカル依存を閉じ込めすぎず、再利用できる部分は共有モジュールへ寄せる

理由:

- `bench/` は再現性と比較可能性が重要
- 実験コードでも、毎回 import や実行方法で迷わない構成の方が価値が高い

## import と実行方法の使い分け

### 絶対 import を使う場面

```python
from sample.google.vertext_ai import TEXT_MODEL
```

向いているケース:

- `sample/` のように直接実行しやすさを優先したいとき
- import 元を明確にしたいとき
- 初学者がコードを追いやすい形にしたいとき

### 相対 import を使う場面

```python
from . import TEXT_MODEL
```

向いているケース:

- package として実行されることが明確なとき
- `demo/` のように module 実行前提で、同一パッケージ内に閉じた依存を表したいとき

注意点:

- 直接ファイル実行では失敗しやすい
- `sample/` では原則避ける

### `python -m ...` を使う場面

```bash
python -m demo.models.model_comparison_demo
```

向いているケース:

- `demo/` のように package として整理された実行物
- 複数ファイル構成で、実行入口を明確にしたいとき
- 相対 import や package 構成を安定して扱いたいとき

### ファイル直接実行を使う場面

```bash
python sample/google/vertext_ai/01_text.py
```

向いているケース:

- `sample/` のような単体サンプル
- まず 1 本だけ素早く試したいとき
- IDE から Current File 実行したいとき

## 判断に迷ったときの優先順位

1. そのコードは `sample`、`demo`、`bench` のどれか
2. 読み手は「学習目的」か「実利用の参考」か「評価・比較目的」か
3. 実行は 1 ファイル単位か、package 単位か
4. その import は初心者が見て迷わないか
5. 共通化すると分かりやすくなるか、逆に見通しが悪くなるか

## 現時点の推奨

- `sample/`: 直接実行しやすい構成、絶対 import 優先
- `demo/`: `python -m ...` 前提、絶対 import 優先
- `bench/`: 再現性重視の CLI、絶対 import 優先

必要になったら個別フォルダに追加の `AGENTS.md` や補助ドキュメントを置いて、さらに細かいルールを定義してよい。
